Monthly Business Review — April 2026

Leaders: Tom Boettner, Liz Mackie & Alex Congleton  |  Domain: Fantasy

📝 Executive Summary

April was Fantasy's most productive engineering month to date. Building on March's foundation, the team shipped end-to-end wallet integration (full debit/credit cycle with error handling and reconciliation), results ingestion and settlement jobs, contest templating and matching infrastructure, and a Kafka outbox pattern for reliable downstream event publishing. The mobile app graduated from a demo environment to live end-to-end communication with the Fantasy API via C4 and gRPC. Quality and launch-readiness work began in earnest: FastLane onboarding accelerated across services, the Engineering Go/No-Go criteria were published, and runbooks were started. The team also made significant product clarity progress, with the Daily Streak design/requirements delivered to Monterosa, key MVP scope documents locked, and cross-team dependency conversations with CAT formalized.

The primary risk entering May is CAT's confidence gap on their August 5 delivery commitment across four features (Unified Wallet, Payments MVP, RG Limits, Geolocation). The Tenant ID architecture decision also remains open and is a dependency for Profile, Payments, and App Switcher.

✅ Successes

Full Wallet Integration Shipped
Tanya Phanich completed end-to-end wallet integration across 10 stories:
  • Happy-path debit on contest entry creation
  • Insufficient-funds and wallet-timeout error handling
  • Automated reconciliation job (runs every 2 min) to refund users if a failure occurs mid-entry
  • WalletTransaction audit table with full ContestEntry relationship
  • Transaction type and fund type combinations mapped to DFS-specific codes (e.g., "DFS cashwin")
Results Ingestion & Settlement Pipeline Online
The full results-to-picks-to-payouts pipeline is now functional:
  • Nigel Young built proposition result ingestion (Kafka consumer), PropositionResult enum, and ResultPickJob to propagate outcomes to all associated picks
  • Salmaan Khan integrated JobRunr into all services and exposed the JobRunr dashboard to ingress for ops visibility
  • Automated background jobs handle scheduling, retries, and recurring result settlement
gRPC API Surface Expanded
Michael Goodwin built out the core API RPCs needed for the mobile experience: League Page, Event Page, My Entries, and a reusable pagination pattern. All RPCs are covered by FAST tests (Patrick Kilburn). The C4 layer is now using gRPC to call the Fantasy API, closing the loop from mobile client → C4 → core API → database.
Kafka Outbox Pattern Implemented
A reliable outbox architecture was shipped to guarantee downstream event publishing (bonus, promotions, etc.) without data loss if a service fails between DB write and Kafka produce. Includes Debezium configuration, outbox table, and proto definitions for contest entry events.
Mobile App: Live End-to-End Data Flow
The Fantasy mobile app (Kotlin/FUEL) is now displaying live data sourced from the Fantasy API via C4, end to end — no hardcoded or mocked data. Jacob Tabak completed the Playbook CMS → C4 → FUEL integration, enabling content managers to update the app's promotional carousel from the same CMS infrastructure as the sportsbook.
Contest Infrastructure: Templates & Matching
Foundation for automated contest lifecycle:
  • Contest and ContestTemplate entities created (Ryan Taylor)
  • Experience Level enum on accounts (Josh Schaefer)
  • Playmaker admin UI supports creating/managing templates and monitoring contests (demo'd in M29.4)
  • M30.1 shipped automated contest creation from templates, dynamic capacity scaling, and state-specific entry rule enforcement
Quality & FastLane Momentum
Patrick Kilburn completed FastLane evaluations for fantasy-api, fantasy-ingestion-worker, and fantasy-c4-api; wrote initial integration tests; created runbooks (published to Confluence); and drafted the Engineering Go/No-Go Launch Criteria with 18 checklist items covering Quality, Security, Regulatory, Platform, and Product sign-offs.
Loyalty Tokens & Offer Integration
Liam Butler completed the foundational loyalty token work: external dependencies mapped, display in app and pickslip, and token consumption/refunding logic. Also wired up the offer-api client and Fantasy Targeting for PBT campaigns — groundwork for acquisition promotions.
Daily Streak Design & Requirements to Monterosa
The Free-to-Play Daily Streak Game design and requirements were delivered to Monterosa by end of April, hitting the deadline required for the August 2026 NFL-season launch. This keeps the FTP acquisition funnel on track.
Product Clarity: Key Docs Landed
Significant product documentation shipped in April, reducing ambiguity for engineering:
  • MVP Scope Commitment locked (Confluence)
  • Unified Wallet Business Requirements & Engineering Brief
  • DFS Deposit Limits for MA and TN
  • Contest Rules & Requirements finalized
  • DFS × Fanatics ONE Tier Point Earn requirements
  • Super App Bootstrap & App Switcher Engineering Spike Response (Andrew Maum)

⚠️ Challenges & Risks

CAT Delivery Confidence Gap
As of April 21, the CAT org stated it is not confident in its ability to deliver by August 5. Four features are dependencies: FEAT-8216 (Universal Balance / Unified Wallet), FEAT-8217 (Payments MVP), FEAT-8509 (RG Limits & Entry Limits), and FEAT-8511 (Geolocation). CAT's M30 prod date is July 15, but DFS needs these integrated and tested by August 5 — that window may be too tight.
Geolocation (FEAT-8511) is particularly uncertain: targeting Sprint 3 or 4 with outstanding OB dependencies.
Tenant ID Architecture Still Open
The Tenant ID decision — whether DFS gets its own tenant ID or shares the FBG tenant with a separate site ID — was still open at the end of April (last noted 4/21). This is a blocker for Profile (App Switcher, KYC), Payments, and CAT work. Liz Mackie is driving the conversation with Griffin Sandler and James Hosken.
Reporting Requirements Unclear
DFS financial reporting requirements remain undefined. Liz Mackie and Alex Congleton were tasked with reaching out to Gabriel R on this. Unresolved reporting requirements risk last-minute scope additions to CAT/payments work.
Story Points Not Consistently Tracked
Sprint velocity data is not available because story points are inconsistently set across FNTSY tickets. This makes burn-rate forecasting difficult as the team approaches the Alpha deadline.
📌 Action Item: Establish a team norm for pointing stories before sprint start. Goal: enable accurate velocity tracking by M30.2.
